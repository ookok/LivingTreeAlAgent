### 3.3 覆盖率目标

| 模块类型 | 覆盖率目标 | 理由 |
|---------|-----------|------|
| 核心引擎（Phase 1） | 90% | 最关键 |
| 智能代理（Phase 2） | 80% | 重要 |

### 3.4 CI/CD 集成

**目标**: 每次 PR 自动运行测试，覆盖率不达标则阻止合并

**工具**:
- GitHub Actions（如果代码在 GitHub）
- 或自行搭建 CI 服务器

**流程**:
1. PR 创建 → 自动运行测试
2. 覆盖率 < 目标 → 标记为 "Changes requested"
3. 开发者补充测试 → 重新运行
4. 覆盖率 ≥ 目标 → 允许合并

## 风险与应对

### 风险 1: 删除代码导致功能失效

**应对**:
1. **逐步删除** - 先标记 `@deprecated`，观察 1-2 天
2. **完整测试** - 删除后运行完整测试套件
3. **快速回滚** - 准备回滚脚本（从 Git 恢复）

### 风险 2: 迁移后导入路径错误

**应对**:
1. **自动化工具** - 使用 `sed` 或 Python 脚本批量更新导入
2. **测试验证** - 运行测试，检查导入错误
3. **手动检查** - 重点检查动态导入（`__import__`, `importlib`）

### 风险 3: 测试覆盖率难以达成

**应对**:
1. **优先级** - 先为核心模块编写测试（高频使用）
2. **测试生成** - 使用 AI 辅助生成测试用例
3. **逐步提升** - 不要求一次性达到 80%，可以逐步提升（如 30% → 50% → 80%）
