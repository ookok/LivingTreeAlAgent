**情绪→思考模式映射规则**：
| 情绪类型 | 强度 | 思考模式 | 理由 |
|---------|------|----------|------|
| 愤怒/焦虑 | > 0.7 | 分析式 | 高强度负面情绪需要理性、专业的回复 |
| 愤怒/焦虑 | ≤ 0.7 | 引导式 | 低强度负面情绪需要引导和支持 |
| 困惑/迷茫/无助 | 任意 | 引导式 | 困惑情绪需要分步骤引导 |
| 喜悦/开心/满意 | 任意 | 沉浸式 | 正面情绪可以使用亲切、有同理心的回复 |
| 中性 | 任意 | 沉浸式（默认） | 保持当前模式 |

**引导式思考模式指令示例**（环评工程师）：
```
【环评工程师引导模式】
在你的思考过程（thinking字段）中，请注重引导和建议，帮助用户理解和决策：

1. **引导式思考**：用"建议..."、"可以考虑..."、"下一步..."等引导性表达
2. **分步骤规划**：将复杂问题分解为简单步骤
3. **思考要素**：
   - 用户当前困惑点识别
   - 分步骤解决方案设计
   - 每个步骤的具体操作指引
   - 可能遇到的问题和应对方法
...
```

## MCP Server 集成方案 ✅ (23:00)

### 问题分析
用户问：**"MCP Server 是否适合作为工具？如果工具功能重复了，如何选择？"**

### MCP Server 适用性分析

**✅ 非常适合**，理由：
1. **标准协议**：MCP（Model Context Protocol）是 Anthropic 推出的标准，Claude/OpenAI 都在支持
2. **零依赖分发**：服务端部署一次，所有客户端即时使用
3. **自动发现**：MCP 标准协议，自动发现工具列表（无需手写 BaseTool 封装）
4. **安全隔离**：服务端沙箱执行，客户端零风险

**重复处理策略**（4 级优先级）：
```
Level 1（首选）：MCP Server 已提供该功能
   → 通过 MCP 协议调用，不重复实现
   → 包装为 MCPToolAdapter（实现 BaseTool 接口）

Level 2（备选）：本地 CLI 工具已安装
   → 通过 CLIToolDiscoverer 发现 + BaseTool 封装

Level 3（降级）：都没有
   → 用 Python 纯算法实现（如 HJ 610 解析法）
   → 或触发 PackageManagerInstaller 自动安装

Level 4（兜底）：
   → 调用云端 API
```

### 实现方案

**新增文件**：`client/src/business/mcp/mcp_tool_adapter.py`
```python
class MCPToolAdapter(BaseTool):
    """将 MCP Server 工具包装为 BaseTool"""
    
    def __init__(self, mcp_tool: dict, server_url: str):
        super().__init__(
            name=mcp_tool["name"],
            description=mcp_tool["description"],
            category="mcp",
            tags=["mcp", mcp_tool.get("server", "")]
        )
        self._server_url = server_url
        self._mcp_tool = mcp_tool
    
    def execute(self, **kwargs) -> ToolResult:
        # 通过 MCP 协议调用远程工具
        import requests
        response = requests.post(
            f"{self._server_url}/tools/call",
            json={"name": self.name, "arguments": kwargs}
        )
        if response.status_code == 200:
            return ToolResult.ok(data=response.json())
        return ToolResult.fail(error=f"MCP call failed: {response.status_code}")
```

**工具选择逻辑**（`tool_registry.py` 扩展）：
```python
async def resolve_tool(tool_name: str) -> BaseTool:
    # 1. 先检查 MCP Server
    mcp_tool = await mcp_registry.find_tool(tool_name)
    if mcp_tool:
        return MCPToolAdapter(mcp_tool, mcp_tool["server_url"])
    
    # 2. 再检查本地工具
    local_tool = tool_registry.find_by_name(tool_name)
    if local_tool:
        return local_tool
    
    # 3. 触发自动安装
    installer = PackageManagerInstaller()
    result = await installer.install_and_register(tool_name)
    ...
```

- **MCP + 本地同时存在** → 优先 MCP（更新、更安全）
- **多个 MCP Server 提供相同工具** → 按优先级配置（用户可调整）
- **本地 CLI 版本更新** → 提示用户升级

## NaturalLanguageToolAdder 实现 ✅ (00:10)

### 文件
- `client/src/business/self_evolution/natural_language_tool_adder.py`（约 700 行）

### 核心能力：自然语言 → 自动安装 + 注册工具

```
用户输入自然语言
    ↓
NaturalLanguageParser（LLM 解析，走 GlobalModelRouter）
    ↓ 输出 ParsedIntent（JSON 结构化）
SourceResolver（解析来源）
    ↓
ToolInstallerAndRegistrar（安装 + 注册）
    ↓
完成：工具可用
```

### 支持的输入类型

| 用户输入示例 | 识别结果 |
|-------------|---------|
| `"帮我安装 pyswmm"` | PyPI 包 → `pip install pyswmm` |
| `"帮我添加 AERMOD 工具"` | 直接名称 → 查预设/扫描 PATH |
| `"npx @modelcontextprotocol/server-filesystem /tmp"` | MCP Server (npm) → `MCPToolAdapter` |
| `"https://github.com/OpenLCALC/openlca-py"` | GitHub 仓库 → clone → 判断类型 → 安装 |
| `"这篇文章提到的工具：https://..."` | 爬取文章 → LLM 提取工具名 → 递归处理 |
| `"下载这个二进制：https://.../tool.exe"` | 下载 → `~/.livingtree/bin/` → 扫描注册 |

### 内部结构

| 类 | 功能 |
|------|------|
| `NaturalLanguageParser` | LLM 解析自然语言 → `ParsedIntent`（走 `GlobalModelRouter`） |
| `SourceResolver` | 根据 `SourceType` 解析来源（GitHub/PyPI/npm/文章/二进制） |
| `ToolInstallerAndRegistrar` | 安装 + 触发 `CLIToolDiscoverer` 扫描 + 注册到 `ToolRegistry` |
| `NaturalLanguageToolAdder` | 主入口，串联以上三步 |
| `add_tool_from_text()` | 便捷函数 |

- ✅ **所有 LLM 调用走 `GlobalModelRouter`**，无直接 Ollama 连接
- ✅ 复用 `PackageManagerInstaller`（pip/npm/cargo/brew/winget/choco）
- ✅ 复用 `CLIToolDiscoverer V2`（增量扫描/安全黑名单/包管理器查询）
- ✅ 复用 `MCPToolAdapter`（MCP Server 接入）
- ✅ LLM 失败时回退到正则解析（`_fallback_parse`）
- ✅ 置信度 < 0.3 时拒绝处理，要求用户提供更详细信息
